# Message Contracts

Audience: developers building channel adapters, providers, or tools that
produce or consume messages in the OSA agent loop.

This document defines the canonical message format that flows through the system
and the Signal Theory classification overlay applied to every message.

---

## Canonical Message Format

A message is a map with the following shape. All messages stored in the database,
passed to LLM providers, and returned from the agent loop conform to this format.

### Schema

```elixir
%{
  # Required
  role:         String.t(),   # "user" | "assistant" | "tool" | "system"
  content:      String.t(),   # Message text. May be empty string for tool-call-only turns.

  # Present on assistant turns that request tool execution
  tool_calls:   list(%{
    id:        String.t(),  # Unique call ID generated by the LLM
    name:      String.t(),  # Tool name as registered in Tools.Registry
    arguments: map()        # Parsed JSON arguments
  }),

  # Present on tool result turns
  tool_call_id: String.t(), # Matches the `id` from the tool_calls entry above

  # Contextual metadata (optional)
  metadata: %{
    session_id:  String.t(),
    channel:     String.t(),  # e.g. "telegram", "cli", "http"
    timestamp:   DateTime.t(),
    token_count: integer() | nil
  }
}
```

### Role Values

| Role | Direction | Description |
|---|---|---|
| `"user"` | inbound | Message from the human user via any channel. |
| `"assistant"` | outbound | Response from the LLM agent. May contain `tool_calls`. |
| `"tool"` | inbound | Result from a tool execution. Paired with `tool_call_id`. |
| `"system"` | setup | System prompt or context injection. Not persisted in the message log. |

### Message Lifecycle

```
User input
    │
    ▼
%{role: "user", content: "List all files in /tmp"}
    │
    ▼  Agent.Loop.process_message/3
    │
    ▼
%{role: "assistant",
  content: "",
  tool_calls: [%{id: "call_1", name: "shell_execute",
                 arguments: %{"command" => "ls /tmp"}}]}
    │
    ▼  Tools.Registry.execute("shell_execute", %{"command" => "ls /tmp"})
    │
    ▼
%{role: "tool",
  content: "file1.txt\nfile2.txt\nreport.pdf",
  tool_call_id: "call_1"}
    │
    ▼  Re-prompt LLM with accumulated messages
    │
    ▼
%{role: "assistant",
  content: "Here are the files in /tmp:\n- file1.txt\n- file2.txt\n- report.pdf",
  tool_calls: []}
```

### Database Representation

Messages are persisted to SQLite via `OptimalSystemAgent.Store.Message`.

```elixir
schema "messages" do
  field :session_id,   :string
  field :role,         :string           # "user" | "assistant" | "tool" | "system"
  field :content,      :text
  field :tool_calls,   :map              # JSON-serialised list of tool_call maps
  field :tool_call_id, :string
  field :channel,      :string
  field :token_count,  :integer
  field :metadata,     :map, default: %{}
  timestamps()                           # inserted_at, updated_at (UTC)
end
```

Content fields are validated as valid UTF-8 on insert. Invalid bytes are
replaced (not rejected) to avoid losing message history.

## Signal Classification Overlay

Every message flowing through the event bus receives a Signal Theory
5-tuple classification. This is an annotation layer on top of the base
message format — it does not replace the canonical message fields.

Signal Theory reference: Luna, R. (2026). *Signal Theory: The Architecture of
Optimal Intent Encoding in Communication Systems.* https://zenodo.org/records/18774174

### The 5-Tuple: S = (Mode, Genre, Type, Format, Weight)

#### Mode

What operational action does this message require?

| Value | Description | Example triggers |
|---|---|---|
| `:execute` | Do something NOW | "run", "send", "deploy", "trigger", "sync" |
| `:build` | Create something new | "create", "generate", "write", "scaffold", "implement" |
| `:analyze` | Produce insight | "analyze", "compare", "metrics", "review", "report" |
| `:maintain` | Fix or update | "fix", "update", "migrate", "debug", "patch", "rollback" |
| `:assist` | Explain or guide | "explain", "how do I", "what is", "help me" |

#### Genre

What is the communicative purpose?

| Value | Description | Example |
|---|---|---|
| `:direct` | A command or instruction | "Deploy to staging." |
| `:inform` | Sharing information | "The build failed at step 3." |
| `:commit` | Making a promise | "I'll handle the migrations." |
| `:decide` | Making or requesting a decision | "Approve this PR?" |
| `:express` | Emotional expression | "Great work!" / "This is broken again." |

#### Type

Domain category of the message:

| Value | Description |
|---|---|
| `"question"` | Asking for information (contains `?`, starts with who/what/when/where/why/how) |
| `"request"` | Asking for an action |
| `"issue"` | Reporting a problem (error, bug, broken, crash, fail) |
| `"scheduling"` | Time-related (remind, schedule, later, tomorrow, next week) |
| `"summary"` | Asking for condensed information (summarize, recap, brief) |
| `"report"` | Providing status or results |
| `"general"` | None of the above |

#### Format

Structural format of the content:

| Value | Description |
|---|---|
| `:text` | Plain prose |
| `:code` | Code block or snippet |
| `:json` | JSON data |
| `:markdown` | Markdown-formatted text |
| `:binary` | Binary/non-text content |

#### Weight

A float in `[0.0, 1.0]` representing signal-to-noise ratio. Computed from
message length and structural complexity.

```
weight = min(length / 500.0, 1.0)
```

The agent loop uses weight as a tool-dispatch gate: messages below `0.20`
receive a plain LLM response with no tool calls. This prevents hallucinated
tool sequences for low-information inputs like "ok" or "lol".

### Signal Struct

```elixir
%MiosaSignal{
  mode:     :execute,
  genre:    :direct,
  type:     "request",
  format:   :text,
  weight:   0.72,
  content:  "Deploy the staging build now",
  metadata: %{}
}
```

### Classification Path

1. **Deterministic** (fast path, no LLM): Pattern-match against known keywords.
   Used for all messages as the base classification. Result confidence: `:low`.

2. **LLM enrichment** (async): For messages routed to the LLM, the provider
   enriches the classification with context-aware reasoning. Result confidence: `:high`.

3. **ETS cache**: Classifications are cached for 10 minutes keyed by content hash.

### Event Bus Integration

Every event emitted via `Events.Bus.emit/3` carries the Signal overlay:

```elixir
{:ok, event} = OptimalSystemAgent.Events.Bus.emit(
  :user_message,
  %{content: "Deploy to production"},
  signal_mode: :execute,
  signal_genre: :direct,
  signal_sn: 0.85
)

event.signal_mode  # => :execute
event.signal_genre # => :direct
event.signal_sn    # => 0.85
```

When `signal_mode` is not set, the bus auto-classifies the event via
`Events.Classifier.auto_classify/1`.

## Provider Message Format

Providers receive messages in the canonical format and must translate to their
native API format internally. The `Providers.Behaviour` contract guarantees
a consistent canonical output regardless of which provider is used.

### Canonical Input (to provider)

```elixir
[
  %{role: "system",    content: "You are OSA, a capable AI assistant..."},
  %{role: "user",      content: "Read config.json"},
  %{role: "assistant", content: "", tool_calls: [
    %{id: "call_1", name: "file_read", arguments: %{"path" => "config.json"}}
  ]},
  %{role: "tool",      content: "{\"port\": 4000}", tool_call_id: "call_1"},
  %{role: "user",      content: "What port is the app running on?"}
]
```

### Canonical Output (from provider)

```elixir
{:ok, %{
  content:    "The application is running on port 4000.",
  tool_calls: []
}}
```

Or, if the LLM wants to call a tool:

```elixir
{:ok, %{
  content:    "",
  tool_calls: [
    %{id: "call_2", name: "shell_execute", arguments: %{"command" => "ps aux | grep node"}}
  ]
}}
```

## Channel Message Contract

Channel adapters deliver inbound messages to the loop using this minimal shape.
The loop enriches with session context before persisting.

```elixir
# Minimum required by Agent.Loop.process_message/3
%{
  role:    "user",
  content: "Your message text here"
}

# With optional channel context
%{
  role:     "user",
  content:  "Your message text here",
  metadata: %{
    channel:   "telegram",
    chat_id:   "123456789",
    username:  "alice"
  }
}
```
